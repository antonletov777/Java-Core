# Исчерпывающее руководство: Многопоточность, Параллелизм и Асинхронность в Java 21

---

## 1. Концепции: Конкурентность, Параллелизм и Асинхронность
*Соответствующий класс: `MultithreadingConceptsLecture1.java`*

Прежде чем писать код, необходимо четко разделить термины, которые часто путают.

### 1.1. Параллелизм (Parallelism)
Это физическое, одновременное выполнение нескольких задач в одну и ту же единицу времени.
* **Многоядерный параллелизм:** Каждое ядро процессора выполняет свой поток. Если у вас 4 ядра, вы можете выполнять 4 задачи *истинно* одновременно.
* **Гипертрейдинг (Hyper-threading):** Технология на уровне железа (например, у Intel), когда одно физическое ядро операционной системой видится как два логических. Это позволяет более эффективно утилизировать такты процессора (пока один поток ждет данных из кэша L3, второй поток на том же ядре выполняет вычисления).

### 1.2. Конкурентность (Concurrency)
Это способность системы поддерживать выполнение нескольких задач в перекрывающиеся периоды времени.
* **Параллелизм на одном ядре (Многозадачность):** Одно ядро физически не может выполнять два потока одновременно. Конкурентность достигается за счет **переключения контекста (Context Switch)**. ОС выделяет каждому потоку квант времени (несколько миллисекунд). Ядро быстро переключается между ними, создавая для пользователя иллюзию одновременной работы.

### 1.3. Асинхронность (Asynchrony)
Это архитектурный подход (модель программирования), при котором инициатор операции не блокируется в ожидании ее завершения.
* В Java асинхронность чаще всего реализуется поверх пулов потоков (например, отдаем задачу в `CompletableFuture`, а текущий поток идет делать другие дела, чтобы потом обработать результат через callback).

## 2. Основы: Жизненный цикл потоков (Thread, Runnable)
*Соответствующий класс: `ThreadBasicsLecture2.java`*

### 2.1. Создание потока
* **Класс `Thread`**: Исторический способ (наследование).
* **Интерфейс `Runnable`**: Современный стандарт бэкенда (композиция). Передается в конструктор `Thread` или в Thread Pool.

### 2.2. Жизненный цикл потока (Thread States)
Поток в Java всегда находится в одном из 6 состояний (`Thread.State`):
1. **NEW**: Поток создан (`new Thread()`), но `start()` еще не вызван.
2. **RUNNABLE**: Поток готов к работе (или уже работает) в JVM, но может ожидать ресурсов ОС (процессорного времени).
3. **BLOCKED**: Поток заблокирован и ждет освобождения монитора (входа в `synchronized` блок).
4. **WAITING**: Поток бесконечно ждет другого потока (вызван `Object.wait()`, `Thread.join()`, `LockSupport.park()`).
5. **TIMED_WAITING**: Поток ждет заданное время (`Thread.sleep()`, `wait(timeout)`).
6. **TERMINATED**: Поток завершил выполнение метода `run()` или упал с ошибкой.

### 2.3. Демон-потоки и Остановка (Daemon, Interrupt)
* **Daemon (Фоновые потоки):** Служебные потоки (например, Garbage Collector). JVM завершает работу, как только заканчиваются все пользовательские (non-daemon) потоки, не дожидаясь завершения демонов.
* **Механизм Interruption (Прерывание):** В Java нельзя "убить" поток принудительно (метод `stop()` удален). Остановка происходит кооперативно: мы ставим потоку флаг `interrupt()`, а сам поток обязан периодически проверять `Thread.currentThread().isInterrupted()` и корректно завершать работу.

## 3. Java Memory Model (JMM) и Примитивы синхронизации
*Соответствующий класс: `JmmAndSyncLecture3.java`*

Процессоры имеют многоуровневый кэш (L1, L2, L3). Когда поток читает переменную, он может скопировать ее в кэш своего ядра. Если другой поток на другом ядре изменит эту переменную, первый поток может этого не увидеть (проблема видимости).

### 3.1. Ключевое слово `volatile`
* Гарантирует **видимость**. Значение переменной всегда читается напрямую из главной памяти (RAM), минуя кэши процессора.
* Запрещает **Reordering** (переупорядочивание инструкций компилятором).
* **ВАЖНО:** `volatile` НЕ гарантирует атомарность операций (например, инкремент `i++` всё равно сломается при конкурентном доступе).

### 3.2. Ключевое слово `synchronized` (Мониторы)
Гарантирует **взаимное исключение (Mutual Exclusion)** и **видимость**.
* У каждого объекта в Java есть скрытый замок — "Монитор".
* Когда поток входит в `synchronized` блок, он захватывает монитор. Любой другой поток, пытающийся зайти в блок с этим же монитором, переходит в состояние `BLOCKED`.
* *Legacy-координация:* Методы `wait()`, `notify()`, `notifyAll()` вызываются **только** внутри `synchronized` блока для передачи сигналов между потоками. В современном бэкенде практически полностью вытеснены классами из пакета `java.util.concurrent`.

### 3.3. Неблокирующая синхронизация (CAS и Atomics)
Пакет `java.util.concurrent.atomic` (`AtomicInteger`, `AtomicReference`).
Основан на аппаратной инструкции **CAS (Compare-And-Swap)**. Это оптимистичная блокировка: мы пытаемся обновить значение, и если никто другой его не изменил за это время, обновление проходит успешно. Работает в сотни раз быстрее `synchronized` в условиях низкой конкуренции.

## 4. Проблемы многопоточного программирования
*Соответствующий класс: `MultithreadingProblemsLecture4.java`*

Многопоточность — это минное поле. Самые частые проблемы:

1. **Race Condition (Состояние гонки):** Ошибка проектирования, когда результат выполнения зависит от того, в каком порядке планировщик ОС выдаст процессорное время потокам. (Например, два потока одновременно делают `balance -= 100`, но списывается только 100 из-за чтения старого значения).
2. **Deadlock (Взаимная блокировка):** Поток А захватил Ресурс 1 и ждет Ресурс 2. Поток Б захватил Ресурс 2 и ждет Ресурс 1. Оба потока зависают навсегда (состояние `BLOCKED`).
3. **Livelock (Взаимная уступка):** Потоки постоянно меняют свое состояние в ответ на действия друг друга, но не делают полезной работы. (Словно два вежливых человека в коридоре постоянно уступают друг другу дорогу, шагая в одну и ту же сторону).
4. **Starvation (Голодание):** Поток с низким приоритетом никогда не получает процессорное время, потому что его постоянно перехватывают более приоритетные (жадные) потоки.

## 5. Продвинутая синхронизация (java.util.concurrent.locks)
*Соответствующий класс: `AdvancedLocksLecture5.java`*

Современная альтернатива `synchronized` с гораздо большими возможностями.

* **`ReentrantLock`**: Повторно входимая блокировка. Главная фича — метод `tryLock()`, позволяющий попытаться захватить замок и уйти, если он занят (предотвращает Deadlock).
* **`ReadWriteLock`**: Позволяет тысячам потоков одновременно читать данные, но эксклюзивно блокирует всех (и читателей, и писателей) при записи.
* **`Condition`**: Современная замена `wait/notify`. Позволяет иметь несколько "очередей ожидания" на одном замке.
* **`Semaphore`**: Ограничивает количество потоков, которые могут одновременно получить доступ к ресурсу (например, ограничение в 50 подключений к БД).
* **`CountDownLatch` / `CyclicBarrier`**: Позволяют потокам дожидаться завершения определенного количества операций другими потоками перед тем, как продолжить работу.

## 6. Thread Pools: Executors и ScheduledExecutorService
*Соответствующий класс: `ExecutorsAndPoolsLecture6.java`*

Создание потока (`new Thread()`) занимает ресурсы ОС (память под стек около 1 МБ, системные вызовы). В бэкенде потоки переиспользуют через Пулы Потоков (Thread Pools).

### 6.1. Фабрика `Executors` (Интерфейс `ExecutorService`)
* `newFixedThreadPool(n)` — Пул фиксированного размера. Классика бэкенда. Лишние задачи копятся в очереди.
* `newCachedThreadPool()` — Пул без ограничений. Создает новый поток, если нет свободных, и убивает потоки, простаивающие более 60 секунд. Опасен `OutOfMemoryError` под нагрузкой.
* `newSingleThreadExecutor()` — Пул из 1 потока. Гарантирует последовательное выполнение задач.
* `ScheduledExecutorService` — Для выполнения задач с задержкой или периодически (раз в минуту). Замена устаревшему `Timer`.

### 6.2. `Callable` и `Future`
В отличие от `Runnable` (`void`), интерфейс `Callable<T>` возвращает результат и может бросить Exception. При отправке `Callable` в пул через метод `submit()`, возвращается объект `Future<T>`. Вызов `future.get()` заблокирует текущий поток до получения результата.

## 7. Асинхронное программирование: CompletableFuture
*Соответствующий класс: `CompletableFutureLecture7.java`*

`Future` имеет недостаток — метод `get()` блокирует поток. `CompletableFuture` (Java 8+) решает эту проблему, позволяя строить асинхронные неблокирующие конвейеры (Пайплайны).

* **Запуск:** `supplyAsync(Supplier)` (с результатом), `runAsync(Runnable)` (без результата).
* **Трансформация:** `thenApply(Function)` (аналог map), `thenAccept(Consumer)` (аналог forEach).
* **Комбинирование:**
    * `thenCompose()` — аналог flatMap (когда вторая задача зависит от результата первой).
    * `thenCombine()` — выполняет две независимые асинхронные задачи параллельно и объединяет их результат.
    * `allOf()` — дожидается завершения массива фьючеров.
* **Обработка ошибок:** `exceptionally()`, `handle()`.

## 8. Эволюция Java 21: Virtual Threads и Structured Concurrency
*Соответствующий класс: `VirtualThreadsAndSpringLecture8.java`*

### 8.1. Virtual Threads (Проект Loom)
Исторически 1 Java-поток (Platform Thread) жестко привязан к 1 потоку ОС. ОС может тянуть максимум ~10 000 потоков. Это проблема "Пропускной способности" (Throughput) в бэкенде: когда поток ждет ответа от БД, он простаивает, но не отдает поток ОС.

* **Виртуальные потоки (`Thread.ofVirtual()`):** Дешевые, легковесные потоки, управляемые самой JVM, а не операционной системой. Их можно создавать **миллионами**.
* **Как работают:** Когда виртуальный поток натыкается на блокирующую I/O операцию (запрос в сеть, ожидание лока), JVM "открепляет" (unmounts) его от потока ОС и отдает поток ОС другому виртуальному потоку. Как только данные приходят, виртуальный поток снова прикрепляется (mounts) и продолжает работу.
* **Итог:** Больше не нужны сложные реактивные фреймворки (WebFlux). Пишем простой синхронный блокирующий код, который под капотом работает невероятно асинхронно.

### 8.2. Интеграция со Spring: Аннотация `@Async`
В Spring Framework асинхронность достигается декларативно:
1. Вешаем `@EnableAsync` на конфигурацию.
2. Создаем бин `Executor` (например, настроенный пул потоков или пул виртуальных потоков в Spring Boot 3.2+).
3. Вешаем `@Async` на метод. Spring оборачивает вызов в прокси: метод мгновенно возвращает управление контроллеру, а сама полезная нагрузка уходит на выполнение в настроенный Thread Pool. Результат метода должен быть завернут в `CompletableFuture`.
