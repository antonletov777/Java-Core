# Исчерпывающее руководство: Работа с сетью в Java

**Цель темы:** Освоение API Java для работы с сетью, умение создавать клиент-серверные приложения и взаимодействовать с внешними ресурсами.
**Результат по теме:** Ты научишься создавать HTTP-серверы, использовать сокеты для обмена данными, выполнять HTTP-запросы и понимать особенности протокола HTTP/2.

## 1. Основы сетевого взаимодействия и Сокеты
*Соответствующий класс: `BlockingSocketLecture1.java`*

Сетевое взаимодействие в Java строится на абстракциях ОС (POSIX API). Базовым элементом является **сокет** — конечная точка для обмена данными между приложениями.

### 1.1. TCP vs UDP
* **TCP (Transmission Control Protocol):** Надежная передача с установкой соединения (как телефонный звонок). Асимметричная модель: есть Сервер (`ServerSocket`), который "слушает", и Клиент (`Socket`), который "звонит".
* **UDP (User Datagram Protocol):** Быстрая передача без гарантии доставки (как отправка писем). Симметричная модель: обе стороны используют один и тот же класс `DatagramSocket`.

### 1.2. Проблема блокирующих вызовов (BIO - Blocking I/O)
Стандартный `ServerSocket.accept()` или чтение из потока блокируют текущий поток (thread) до тех пор, пока не появятся данные.
* **Минус:** Приходится создавать новый поток ОС на каждое клиентское соединение ("один поток — одно соединение").
* **Итог:** При тысячах одновременных клиентов сервер быстро исчерпает ресурсы (память на стеки потоков).

## 2. Неблокирующий ввод-вывод (Java NIO)
*Соответствующий класс: `NioSocketLecture2.java`*

Для решения проблемы масштабируемости в Java добавили пакет `java.nio`.

### 2.1. Основные компоненты NIO
* **Каналы (Channels):** Двусторонние пути для передачи данных (например, `SocketChannel`, `ServerSocketChannel`).
* **Буферы (Buffers):** Контейнеры для данных в памяти, через которые каналы читают и пишут.
* **Селекторы (Selectors):** Механизм мультиплексирования. Один поток может следить за событиями (подключение, готовность к чтению/записи) сразу в тысячах каналов. Под капотом использует эффективные системные вызовы (например, `epoll` в Linux со сложностью $O(1)$).

### 2.2. Сравнение BIO и NIO
| Параметр               | Блокирующий I/O (BIO)               | Неблокирующий NIO                   |
|:-----------------------|:------------------------------------|:------------------------------------|
| **Модель потоков**     | 1 поток = 1 соединение              | 1 поток = множество соединений      |
| **Масштабируемость**   | Ограничена числом потоков           | Ограничена ресурсами системы        |
| **Потребление памяти** | Высокое (стек потоков)              | Низкое (буферы)                     |
| **Идеальный случай**   | Немного соединений, долгие операции | Много соединений, короткие операции |

## 3. Эволюция протокола HTTP
HTTP — это протокол прикладного уровня, работающий по модели "Request-Response" (Запрос-Ответ).

### 3.1. Структура сообщений
* **Запрос (Request):** Метод (GET, POST, PUT, DELETE), Путь, Версия протокола + Заголовки + Пустая строка + Тело.
* **Ответ (Response):** Версия, Статус-код (200, 404, 500) + Заголовки + Пустая строка + Тело.

### 3.2. Версии HTTP
* **HTTP/1.1 (1999 г.):** Текстовый протокол. Главная проблема — Head-of-Line (HOL) Blocking (Блокировка в начале очереди). Запросы в одном соединении обрабатываются строго последовательно. Приходилось открывать по 6-8 соединений к серверу для параллельной загрузки.
* **HTTP/2 (2015 г.):** Бинарный протокол (в 2-5 раз меньше служебных данных). Решил проблему ограничений соединений с помощью **мультиплексирования** — все ресурсы передаются параллельно внутри одного TCP-соединения.
* **HTTP/3 (2022 г.):** Перешел с TCP на QUIC (поверх UDP). Это полностью устранило HOL Blocking на уровне транспорта (потеря пакета больше не блокирует остальные потоки данных), ускорило установку соединения (0-1 RTT) и встроило шифрование TLS 1.3 на уровень протокола.

## 4. Современный HTTP-клиент в Java
*Соответствующий класс: `ModernHttpClientLecture3.java`*

С выходом Java 11 в стандарте появился новый пакет `java.net.http`. Он пришел на смену устаревшему `HttpURLConnection`.
* Поддерживает HTTP/1.1 и HTTP/2 "из коробки".
* Позволяет выполнять как синхронные, так и полностью асинхронные запросы (возвращает `CompletableFuture`).
* Реализует удобный паттерн Builder для настройки клиента (`HttpClient.newBuilder()`) и самого запроса (`HttpRequest.newBuilder()`).
