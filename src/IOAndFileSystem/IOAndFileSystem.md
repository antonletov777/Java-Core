## 1. Иерархия потоков: «Матрешки» из байтовых и символьных (Streams)
*Соответствующий класс: `StreamsHierarchyLecture1.java`*

Архитектура `java.io` построена на паттерне **Decorator (Декоратор)** — это принцип «матрешки». Есть базовые классы-источники (читают/пишут напрямую в файл или память) и классы-обертки (добавляют функционал). IO-операции всегда выходят за пределы оперативной памяти.



### 1.1. Байтовые потоки (`InputStream` / `OutputStream`)
Работают с сырыми 8-битными бинарными данными (изображения, архивы, байт-код).
* **Источники (Sources):**
    * `FileInputStream` / `FileOutputStream` — прямое чтение и запись в файлы на диске.
    * `ByteArrayInputStream` / `ByteArrayOutputStream` — чтение/запись сырых байтов в массив в оперативной памяти (часто используется для мокирования I/O в тестах).
    * `PipedInputStream` / `PipedOutputStream` — межпотоковое взаимодействие (передача байт от одного Thread к другому напрямую).
* **Декораторы/Обертки (Wrappers):**
    * `BufferedInputStream` / `BufferedOutputStream` — добавляют буферизацию для снижения системных вызовов.
    * `DataInputStream` / `DataOutputStream` — позволяют читать/писать примитивы Java (`readInt()`, `writeDouble()`) в бинарном формате.
    * `ObjectInputStream` / `ObjectOutputStream` — обеспечивают механизм сериализации объектов.
    * `PrintStream` — обертка для удобного вывода (именно им является `System.out`).

### 1.2. Символьные потоки (`Reader` / `Writer`)
Работают с 16-битными символами `char`. Оптимизированы для текста и автоматически учитывают кодировки (UTF-8, Windows-1251).
* **Источники (Sources):**
    * `FileReader` / `FileWriter` — чтение/запись текстовых файлов.
    * `CharArrayReader` / `CharArrayWriter` — чтение/запись в массив символов в памяти.
    * `StringReader` / `StringWriter` — чтение/запись строк в памяти.
    * `PipedReader` / `PipedWriter` — передача текста между потоками выполнения.
* **Декораторы/Обертки (Wrappers):**
    * `BufferedReader` / `BufferedWriter` — буферизация текста. `BufferedReader` предоставляет важнейший метод `readLine()`.
    * `PrintWriter` — символьный аналог `PrintStream`, самый удобный инструмент для форматированной записи логов и отчетов (`println`, `printf`).

### 1.3. Классы-мосты (Bridges)
Транслируют сырые байты в символы и обратно. Критически важно при получении бинарного потока (из сети или консоли), содержащего текст.
* `InputStreamReader` — оборачивает `InputStream` в `Reader`.
* `OutputStreamWriter` — оборачивает `OutputStream` в `Writer`.
* *Пример:* `new BufferedReader(new InputStreamReader(System.in, StandardCharsets.UTF_8))` — классический способ правильно собрать «матрешку» для чтения текста из консоли.

## 2. Буферизация и Запись данных
*Соответствующий класс: `BufferedIOLecture2.java`*

**Главное правило I/O:** Чтение или запись по одному байту напрямую на диск — "смерть" для производительности из-за накладных расходов на системные вызовы (System Calls) к ядру ОС.

* **Механика буферизации:** Классы-обертки (`BufferedInputStream`, `BufferedWriter`) собирают данные в массиве в памяти JVM. Физическое обращение к диску происходит только когда буфер заполнен или вызван метод `flush()`. Буферизация ускоряет работу в сотни раз.
* **Закрытие ресурсов:** Обязательно использование конструкции `try-with-resources`. При вызове неявного `close()`, буфер автоматически вызовет `flush()`, гарантируя, что остатки данных будут записаны на диск.

## 3. Базовый Ввод/Вывод: Стандартные потоки и Консоль
*Соответствующий класс: `StandardIOLecture3.java`*

В Java есть три стандартных потока, связанных с консолью операционной системы:
* `System.in` — стандартный поток ввода (по умолчанию клавиатура).
* `System.out` — стандартный поток вывода (консоль).
* `System.err` — стандартный поток ошибок (консоль, но часто выводится красным цветом или перенаправляется в логи).

**Класс `java.io.Console`**
В реальной разработке (например, при написании CLI-утилит) чтение паролей через стандартный `System.in` небезопасно. Для этого используется класс `Console`:
* Читает пароль без отображения символов на экране (`console.readPassword()`).
* Возвращает пароль в виде `char[]`, а не `String`, чтобы разработчик мог вручную затереть его в памяти (заполнить нулями) сразу после использования.

## 4. Чтение данных: Удобство vs Производительность (FastScanner)
*Соответствующий класс: `ParsingPerformanceLecture4.java`*



Один из самых частых вопросов на практике — чем и как читать данные.

### 4.1. Класс `Scanner` (Удобство)
Идеален для простых задач, парсинга текста и небольших файлов.
* **Плюсы:** Умеет парсить примитивы "на лету" (`nextInt()`, `nextDouble()`), поддерживает регулярные выражения и разделители.
* **Минусы:** **Катастрофически медленный.** `Scanner` имеет маленький размер буфера (1 КБ) и тратит огромное количество процессорного времени на разбор регулярных выражений и синхронизацию. Не подходит для HighLoad и олимпиадного программирования.

### 4.2. `BufferedReader` + `StringTokenizer` (Fast I/O)
Стандарт де-факто для высоконагруженных систем и парсинга гигантских логов.
* **Плюсы:** Буфер по умолчанию 8 КБ (можно увеличить). Читает текст сырыми блоками, не тратя время на регулярные выражения.
* **Практика (Fast I/O):** Внедрение паттерна `FastScanner`. Данные читаются строками через `br.readLine()`, а затем разбиваются на токены с помощью легковесного `StringTokenizer` (или метода `String.split()`, хотя он медленнее из-за создания новых массивов).

## 5. Современная работа с файлами (NIO.2)
*Соответствующий класс: `Nio2FilesPathsLecture5.java`*

В Java 7 появился пакет `java.nio.file`. Старый класс `java.io.File` считается устаревшим (плохо работает с ошибками, зависает на огромных директориях).

### 5.1. Paths и Files
* `Path` — современный интерфейс для путей. Позволяет нормализовать пути (удалять `..`), объединять их (`resolve()`).
* `Files` — мощнейший утилитный класс, заменяющий рутинное написание потоков:
    * `Files.readString(path)` — прочитать весь небольшой файл в одну строку (Java 11+).
    * `Files.readAllLines(path)` — прочитать файл в `List<String>`.

### 5.2. Чтение гигантских файлов (Stream API)
Если файл весит 50 ГБ, его нельзя загрузить в память через `readAllLines`.
* `Files.lines(path)` — возвращает ленивый `Stream<String>`. Файл читается построчно, память не переполняется. Обязательно требует закрытия (через `try-with-resources`).

### 5.3. Обход директорий (FileVisitor)
Для поиска файлов или удаления целых деревьев папок используется `Files.walkFileTree()`. Он применяет паттерн Visitor, что исключает риск `StackOverflowError` (в отличие от старой рекурсии через `File.listFiles()`).

## 6. Мониторинг изменений (WatchService)
*Соответствующий класс: `WatchServiceLecture6.java`*

* **Суть:** API для отслеживания изменений в папках. Избавляет от необходимости писать вечный цикл `while(true)`, который "сжигает" процессор (busy-waiting).
* **События:** Реагирует на создание (`ENTRY_CREATE`), удаление (`ENTRY_DELETE`) и модификацию файлов (`ENTRY_MODIFY`). Работает поверх нативных системных уведомлений (inotify, FSEvents).

## 7. Сериализация: Классика и Реальность
*Соответствующий класс: `SerializationLecture7.java`*

* **Нативная сериализация Java:** Превращение объекта в байты через `ObjectOutputStream`. Требует маркерного интерфейса `Serializable`. `transient` исключает поле из сохранения.
* **Суровая реальность Backend'а:** Нативную сериализацию Java **не используют** в современном мире. Она имеет критические уязвимости безопасности (вектор атаки RCE при десериализации), работает медленно и не является кросс-языковой.
* **Альтернативы:** В реальных проектах объекты сериализуют в **JSON** (Jackson/Gson) или бинарные форматы **Protocol Buffers (gRPC)** / **Avro**.

## 8. «На десерт»: Производительный I/O: Java NIO (New I/O)
*Соответствующий класс: `NioCoreLecture8.java`*

NIO предлагает буфер-ориентированный и неблокирующий подход (Non-blocking I/O). Это фундамент для написания высокопроизводительных серверов.



### 8.1. Каналы и Буферы
* **Buffers (`ByteBuffer`):** Фундаментальный контейнер.
    * *Heap ByteBuffer:* Выделяется в куче JVM. Быстро создается, но требует копирования данных при передаче в ОС.
    * *Direct ByteBuffer:* Выделяется вне кучи (прямо в памяти ОС). Долго создается, но дает доступ "Zero-Copy" (нет копирования данных в JVM). Идеально для HighLoad сетей и баз данных.
* **Channels (`FileChannel`, `SocketChannel`):** Двунаправленные магистрали для асинхронной передачи данных напрямую в буферы.
* **RandomAccessFile:** Позволяет прыгать по файлу вперед-назад (метод `seek()`) и одновременно читать/писать. Часто используется вместе с `FileChannel` для многопоточной загрузки файлов.

### 8.2. Селекторы (Selectors)
Решение "Проблемы 10K соединений" для серверов.
* **Мультиплексирование:** Один поток (Thread) с помощью `Selector` может отслеживать тысячи неблокирующих каналов одновременно (ожидая событий `OP_READ`, `OP_ACCEPT`). Базируется на эффективных механизмах ОС (epoll в Linux). Это ядро серверов Netty, Tomcat, Nginx.
